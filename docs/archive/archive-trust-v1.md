# Trust Manifest Design

## Problem

CryptoSkill currently scores skills on documentation quality, security patterns, and depth — but not on **trust assumptions**. A user installing a skill that says "swap on Uniswap" today has no way to know:

- Is the code talking only to Ethereum RPC, or also to a centralized backend that could rug?
- If the skill author's API server disappears, do my funds become unrecoverable?
- Does this skill need a 6/8 multisig admin to be honest, or is signing always client-side?
- Has anyone outside the skill author actually reviewed this code?

Existing scoring (npm-style quality, security regex scans) catches malware but not these structural trust questions. This is the gap.

## Design Goals

1. **Machine-readable ingredients** — every external dependency a skill calls (URLs, contracts, RPCs) is enumerated in a `TRUST.md` file we can render and compare.
2. **Three binary trust dimensions** — `audited`, `local-first`, `trustless`. Each is yes / no / unknown with citations.
3. **Auditor attribution** — for each trust claim, show who said so (skill author / CryptoSkill team / third-party auditor / on-chain attestation).
4. **Skin-in-the-game reputation** — borrow ERC-8004 staking primitives so attestors slash if proven wrong.
5. **Inspectable in the UI** — every skill page shows its file tree, raw SKILL.md, and ingredient list.

The output is a "nutrition label" for skill trust: legible to a non-expert, verifiable by an expert.

## Non-Goals

- **Proving correctness** — we are not building Certora/formal verification. We surface trust assumptions; experts still review.
- **Centralized arbitration** — disputes resolve via on-chain attestation, not CryptoSkill staff judgment.
- **Trust scoring as a single number** — quality has one score (existing), trust has a manifest. They are orthogonal.

## TRUST.md Schema

Every skill gets a sibling `TRUST.md` next to `SKILL.md` and `SOURCE.md`. Auto-generated by static analysis, manually edited for accuracy.

```yaml
---
schema_version: 1
skill: binance-spot-api
generated_at: 2026-04-26T00:00:00Z
generated_by: cryptoskill-bot/1.0

# === Three binary dimensions ===

audited:
  status: yes  # yes | no | unknown
  attestors:
    - kind: third_party
      name: Trail of Bits
      url: https://github.com/trailofbits/audits/binance-spot-api-2026.pdf
      date: 2026-03-15
      scope: full

local_first:
  status: no  # yes if only user-specified endpoints, no if any default external
  reason: |
    Skill defaults to api.binance.com. Routes order signing through
    Binance's centralized API — there is no local-only mode.

trustless:
  status: no
  reason: |
    Order custody is on Binance's centralized exchange. If Binance
    halts withdrawals, funds are inaccessible regardless of local code.

# === Ingredient list ===

ingredients:
  - kind: api_endpoint
    host: api.binance.com
    purpose: Order placement, market data, account queries
    trust_label: trusted_operator  # See trust_label taxonomy below
    operator: Binance Holdings Ltd
    hardcoded: true  # User cannot point this elsewhere
    failure_mode: |
      If api.binance.com is offline or returns malicious data, all
      operations fail. There is no fallback.

  - kind: smart_contract
    chain: ethereum
    address: "0x..."
    purpose: ...
    upgradeable: true
    admin_type: multisig_4_of_7
    audit: ...

  - kind: rpc_provider
    default: api.binance.com
    user_overridable: false
    purpose: ...

  - kind: bridge
    name: Arbitrum canonical bridge
    l2beat_stage: 1
    withdrawal_delay: 7d
    council_size: 6_of_9
    used_for: ...

# === Skin-in-the-game (Phase 4) ===

attestations:
  - attestor: 0xerc8004_agent_id
    stake_wei: 1000000000000000000
    claim: "audited:yes"
    expires: 2026-10-26T00:00:00Z
    challenge_window: 14d
---

# Free-form notes from the skill author or auditor below the YAML.
```

### Trust Label Taxonomy (for ingredients)

| Label | Meaning | Example |
|---|---|---|
| `decentralized` | No single party can censor or steal | Ethereum L1, IPFS pinning if multi-pinner |
| `user_specified` | User must provide their own endpoint | RPC URL the user fills in |
| `multisig_controlled` | A small group can change behavior | Most L2 bridges, upgradeable contracts |
| `trusted_operator` | Single party can change/halt | CEX APIs, centralized RPC providers, project-run backends |
| `decentralized_with_council` | Decentralized but with emergency council | OP Stack stage 1 chains |
| `unknown` | We could not classify it | Default for new ingredients pending review |

### Ingredient Kinds

- `api_endpoint` — HTTP API
- `smart_contract` — On-chain contract address
- `rpc_provider` — JSON-RPC node
- `bridge` — Cross-chain bridge
- `oracle` — Price feed (Chainlink, Pyth, etc.)
- `relayer` — Off-chain message router
- `signer` — External signer (custodial wallet, MPC service)
- `data_source` — Indexer or analytics API (Dune, The Graph)
- `library` — npm/pip package the skill installs at runtime

## Three Trust Dimensions — Decision Rules

### `audited`

| Status | Rule |
|---|---|
| `yes` | A named auditor with a public report has reviewed the SKILL.md and any scripts. Report URL must be reachable. |
| `no` | No third-party audit. Only skill author or CryptoSkill bot has reviewed. |
| `unknown` | Audit claimed but URL unreachable or scope unclear. |

CryptoSkill team's own review counts as `no` for this dimension — we are not a third party for our own registry. Self-attestation by skill authors counts as `no`.

### `local_first`

A skill is local-first if every external endpoint can be overridden by the user, OR every default endpoint is decentralized infrastructure (Ethereum L1 RPC where the user can swap providers, IPFS, etc.).

| Status | Rule |
|---|---|
| `yes` | All `api_endpoint` ingredients are either user-specified or decentralized. No hardcoded `trusted_operator` calls. |
| `no` | At least one ingredient is `trusted_operator` and `hardcoded: true`. |
| `unknown` | Ingredient list incomplete. |

### `trustless`

A skill is trustless if user funds remain recoverable when every `trusted_operator` and `multisig_controlled` ingredient becomes hostile or disappears.

| Status | Rule |
|---|---|
| `yes` | All signing happens client-side, no custody at any step, all on-chain interactions go through stage 2 systems or L1, all admin-upgradeable contracts are exit-able. |
| `no` | Any ingredient is `trusted_operator` for custody/signing, OR any contract is upgradeable without exit. |
| `unknown` | Pending classification. |

A CEX trading skill is `trustless: no`. A self-custody Uniswap swap skill that takes a user RPC and only signs locally is `trustless: yes`. A Uniswap swap skill that uses Alchemy's hardcoded RPC is `trustless: no` (Alchemy can censor).

## Auto-Extraction Pipeline

Phase 1 generates `TRUST.md` automatically:

1. **URL extraction** — regex scan SKILL.md, references/, scripts/ for HTTPS URLs and contract addresses.
2. **Classification** — match each URL against a known-host table:
   - `*.binance.com` → `trusted_operator` (operator: Binance)
   - `mainnet.infura.io` → `trusted_operator` (operator: ConsenSys)
   - `api.alchemy.com` → `trusted_operator` (operator: Alchemy)
   - `api.coingecko.com` → `trusted_operator` (operator: CoinGecko)
   - `0x...` (verified Uniswap router) → `decentralized` (immutable)
   - `0x...` (Arbitrum bridge) → `multisig_controlled` (stage 1)
   - User RPC placeholder (`{RPC_URL}`, `${ETH_RPC}`) → `user_specified`

   We maintain `scripts/trust-labels.json` mapping ~500 known hosts and contract addresses. Bot updates from L2BEAT and DefiLlama on the 6h cycle.

3. **Dimension inference** — apply the decision rules above to the populated ingredient list.

4. **Pre-fill `audited: no`** — phase 1 leaves `audited` set to `no` (no third-party audit). Authors and reviewers can edit.

5. **Hash and lock** — hash the SKILL.md + scripts. If hash changes, mark TRUST.md stale until regenerated.

## Auditor Attribution Model

Every trust claim has a list of attestors:

```yaml
audited:
  status: yes
  attestors:
    - kind: third_party
      name: Trail of Bits
      url: https://github.com/trailofbits/audits/...
```

`kind` enum:
- `self` — skill author asserts (lowest trust, default for unverified projects)
- `cryptoskill_team` — our manual review (medium trust)
- `community` — non-affiliated reviewer with PR receipts (medium trust)
- `third_party` — named auditor with public report (high trust)
- `onchain_attestation` — ERC-8004 attestation with stake (high trust, slashable)

A skill can have multiple attestors per dimension. The UI shows all of them.

## ERC-8004 TCR Layer (Phase 4)

Borrow Mike Goldin's TCR mechanic but anchor identities in ERC-8004 agent IDs:

1. **Attestation** — an ERC-8004 agent stakes ETH (or USDC) to claim "skill X has property Y" with an expiry.
2. **Challenge window** — 14 days. Anyone can challenge by staking equal amount + posting evidence.
3. **Resolution** — challenges resolve via Kleros-style jury OR via deterministic on-chain check (e.g. for `local_first`, an automated re-scan of the SKILL.md vs the network calls in scripts/).
4. **Slashing** — losing side's stake goes to the winner minus a registry fee.

This means trust claims have skin in the game. A reckless attestor loses real money.

Phase 4 is gated on Phase 1-3 maturity. We need a clean ingredient list and a working manual attribution flow before we add staking.

```
|-------------|-------------------|-------------------|----------|
| api_endpoint| api.binance.com   | trusted_operator  | Binance  |
| ...

== Source ==
[ View SKILL.md ]  [ View file tree ]  [ Open on GitHub ]
```

### File browser

Use the GitHub raw API plus `marked` for inline rendering. No iframe — render markdown directly into a modal or expanding panel for fast load. File tree comes from the existing repo structure (already known at build time).

### Trust filter on the registry

Add filter pills: `Audited only` · `Local-first only` · `Trustless only`. Defaults to off. Combinable with category filter.

## Implementation Phases

| Phase | Deliverable | Effort |
|---|---|---|
| **1** | `TRUST.md` schema, auto-extractor, `trust-labels.json` for top 200 hosts | 2-3 days |
| **2** | Trust badges on skill cards, ingredient table on skill pages | 1-2 days |
| **3** | Manual attribution pipeline: how to mark a skill as audited via PR | 1 day |
| **4** | File viewer in UI (SKILL.md modal, file tree) | 1 day |
| **5** | Trust filter on main registry | half day |
| **6** | ERC-8004 attestation contract + UI for staked claims | 1-2 weeks |

Ship 1-5 first. They give immediate value with no on-chain dependencies. Phase 6 only after we have ≥50 manual attestations and clear demand.

## Risks and Open Questions

1. **Trust label drift** — projects upgrade contracts, change admin keys, rotate RPC providers. Our static mapping goes stale. Mitigation: bot rescans every 6h, surfaces "trust label changed" alerts.

2. **Bypass via redirects** — a skill could declare `api.coingecko.com` but actually call a wrapper. Mitigation: phase 6 introduces runtime attestation via signed network logs, but phase 1 trusts the static analysis.

3. **TCR death spiral** — historical TCRs (adchain, foam) failed because curators had no recurring incentive to maintain registry quality after the initial token sale. Mitigation: don't issue a new token. Stake in ETH/USDC. Pay attestors a portion of the registry fee from the website (which we charge for premium skills, if we ever do).

4. **Audit centralization** — if only Trail of Bits and OpenZeppelin can audit, we recreate the gatekeeper problem. Mitigation: support `community` attestor kind, weight by reputation but allow many.

5. **Ingredient explosion** — a skill that uses 30 contracts has 30 ingredients. UX needs grouping. Mitigation: collapsible by ingredient kind in the UI; show top 5 by default.

## Comparison to Prior Art

| System | What we borrow | What we change |
|---|---|---|
| **L2BEAT** | Stage 0/1/2 classification of trust assumptions | Apply same taxonomy to skill ingredients, not just L2s |
| **SBOM (CycloneDX/SLSA)** | Machine-readable dependency list | Label each dep with trust kind, not just version |
| **Mike Goldin TCRs** | Stake to attest, challenge to remove | Anchor identity in ERC-8004 instead of fresh token |
| **GoPlus AgentGuard** | Tiered trust registry (Untrusted/Restricted/Trusted) | Three orthogonal binary dimensions instead of one tier |
| **DeFiSafety** | Manual scoring with documentation | Automate static parts; manual only for `audited` |
| **Local-first software (Ink & Switch)** | The local-first concept | Adapted to "all ingredients user-overridable or decentralized" |

## Out-of-Band Items

These don't fit cleanly in this doc but are necessary for the full vision:

- **Trust delta over time** — show when a skill went from `trustless: yes` to `trustless: no` (e.g. operator changed admin keys). JSONL history per skill.
- **Cross-skill ingredient view** — "which skills depend on Infura?" Useful for users who want to filter out a single point of failure.
- **Reputational decay** — old audits (>1 year) auto-downgrade. Re-audit refreshes.
- **API for AI agents** — expose `/api/trust/{skill_id}` so agents can refuse to install skills failing user-defined trust thresholds.
